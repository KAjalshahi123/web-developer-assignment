# Task 03 — Integration Design (OrthoNow → HubSpot / WhatsApp / Google Ads)

**Architecture:**
On submit, the landing page form calls a small serverless function (Vercel/Cloudflare Function) via `fetch()` — not a native HubSpot embed, and not Zapier/Make. A native HubSpot embed would hand over full design control to HubSpot's form renderer, which conflicts with the custom-built page in Task 2. Zapier/Make is a reasonable MVP but adds a second hop of latency and its own failure surface, which is risky against a hard 2-minute WhatsApp SLA — a direct API call gives me control over retries, timeouts, and execution order that a black-box automation tool doesn't.

The serverless function does three things in parallel, not sequentially: (1) calls the **HubSpot Contacts Search API** first, querying on a normalized `phone_number` custom property, then either updates the matched contact or creates a new one via the **Contacts API**, setting Clinic Preference, Source = "Google Ads - Consultation Landing Page", and Lead Status = "New Enquiry"; (2) calls **Karix's WhatsApp Business API** directly to send the confirmation; (3) the front-end, independently of the backend call, fires the `consultation_form_submitted` GTM event, which a GTM tag maps to the Google Ads conversion — this doesn't need to wait on the CRM write at all, so it's never at risk from a slow backend.

**Biggest failure point:** HubSpot's default deduplication key is **email**, not phone — and this form never collects email. Without a fix, every submission creates a *new* contact, even from returning patients. Fix: create a custom unique/indexed contact property `phone_normalized` (E.164 format), and always search-before-create on that property via the Search API.

This also exposes the actual trap in this task: if two different patients submit with the *same* phone number (a shared household landline, a common scenario in Indian lead gen), a naive "update on match" logic would silently overwrite the first patient's name with the second's. My fix: don't blind-overwrite the `name` field — instead, log every submission as a timeline note/engagement on the existing contact (keeping full history), and flag the contact for manual review if the name on a new submission doesn't match the stored name.

**SLA monitoring:** The WhatsApp leg can break from Karix rate limits, serverless cold starts, or a slow HubSpot write blocking the WhatsApp call if run sequentially — which is exactly why they run in parallel. I'd add a lightweight health-check cron hitting the function every few minutes and a Slack alert if WhatsApp latency exceeds 90 seconds or the error rate crosses a threshold, so failures are caught before they become a pattern rather than after a patient complains.
